# Media Optimization: AVIF/AV1 Conversion Pipeline

## Summary

Add a backend media conversion pipeline that optimizes user-uploaded images to AVIF and videos to AV1 (MKV container). The conversion runs asynchronously via an in-process queue. Uploaders see a 7.css spinner while conversion completes; other users only see the final optimized file delivered normally via WebSocket.

## Architecture Overview

```
User A uploads media → upload endpoint saves original → returns original URL
       ↓
Message sent with original image_url → optimistic local render for User A (spinner overlay)
       ↓
Conversion queue picks up file → ffmpeg converts to AVIF/AV1
       ↓
On completion:
  - DB message.image_url updated to optimized path
  - Original file deleted
  - WS `chat_message` broadcast to ALL users with optimized URL
  - Uploader's pending message replaced (spinner gone); others see it for the first time
```

### 2. Backend — MediaManager service

**New file:** `backend/services/media_manager.py`

A singleton class modeled after the existing `db_writer.py` queue pattern:

```python
class MediaManager:
    def __init__(self):
        self._queue: asyncio.Queue | None = None
        self._task: asyncio.Task | None = None

    def start(self):
        """Called from app lifespan, like start_writer()"""
        ...

    async def enqueue(self, original_path: str, message_id: str, author_id: str):
        """Queue a file for conversion. Non-blocking."""
        ...

    async def _worker_loop(self):
        """Pull jobs from queue, run ffmpeg in thread executor."""
        ...

    async def _convert_image(self, input_path: str) -> str | None:
        """ffmpeg -i input.png -c:v libaom-av1 -crf 30 -still-picture output.avif"""
        ...

    async def _convert_video(self, input_path: str) -> str | None:
        """ffmpeg -i input.mp4 -c:v libaom-av1 -crf 35 -c:a libopus output.mkv"""
        ...
```

Key details:
- Uses `asyncio.get_event_loop().run_in_executor(None, subprocess.run, ...)` to avoid blocking the event loop
- AVIF quality: CRF 30 (good balance of size/quality for chat images)
- AV1 quality: CRF 35, audio re-encoded as opus
- On success: original file deleted, DB updated via `enqueue_write`, WebSocket notifications sent
- On failure: log error, leave original file as-is (graceful degradation)
- Detect media type by extension/magic bytes

**Module-level singleton:** `media_manager = MediaManager()`

### 4. Backend — Hook conversion into message send flow

**File:** `backend/api/chat.py`

In `send_message()`, the logic branches based on whether there's media:

```python
has_media = bool(result.get("image_url"))

if not has_media:
    # Text-only: broadcast immediately (existing behavior)
    await ws_manager.broadcast({"type": "chat_message", "data": result})
else:
    # Media: enqueue for conversion, broadcast deferred to media_manager
    filepath = os.path.join(UPLOAD_DIR, os.path.basename(result["image_url"]))
    if os.path.exists(filepath):
        await media_manager.enqueue(filepath, result["id"], author_id)
```

No WS broadcast happens for media messages at this point. The broadcast is deferred until conversion completes (or fails).

### 5. Backend — Avatar conversion

**File:** `backend/api/upload.py`

In `upload_avatar()`, after saving the avatar file, enqueue it for conversion via the same `media_manager` queue.

Avatar-specific considerations:
- Avatar filenames are `{user_id}.ext` — after conversion, becomes `{user_id}.avif`
- Need to delete the original ext version and update DB `avatar_url`
- On completion: broadcast `user_updated` with new avatar URL (existing WS event, same as current avatar update flow)

### 6. Backend — WebSocket events for conversion completion

**File:** `backend/ws/manager.py` (no changes needed, existing `send_to` and `broadcast` suffice)

The `media_manager` emits a single `chat_message` broadcast to ALL users (including the uploader) once conversion completes. No separate `media_optimized` event needed — the frontend handles deduplication.

**Message flow for media messages:**

1. `send_message()` saves to DB with original `image_url`, returns HTTP 201 to uploader — **does NOT broadcast**
2. Uploader's client optimistically adds message to local state with `pending: true` → shows spinner
3. Other users see nothing — transparent
4. Conversion completes → `media_manager`:
   - Updates DB `message.image_url` to optimized path
   - Deletes original file
   - Broadcasts `chat_message` to ALL users with optimized URL
5. Uploader's client replaces the pending message (spinner gone); other users see message appear for the first time

**Message flow for text-only messages** (unchanged):
- `send_message()` saves + broadcasts immediately (no conversion)

**Fallback if conversion fails:**
- Original file kept as-is
- Broadcast `chat_message` with original URL so message isn't lost
- Log error

## Key Design Decisions

1. **ffmpeg subprocess** — no extra Python deps, handles both AVIF and AV1
2. **asyncio Queue** — consistent with existing `db_writer.py` pattern, runs ffmpeg in thread executor
3. **Optimistic UI** — uploader sees message immediately with spinner, others see nothing until conversion done
4. **Graceful degradation** — if conversion fails, original file is served as-is
5. **Original cleanup** — original files deleted after successful conversion to save disk space
6. **No new Python dependencies** — ffmpeg called via `subprocess.run`
